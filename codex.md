# 每日生活记录 App — Codex 限制性编程规则

> 本文件用于约束 Codex 在本项目中的开发行为。  
> Codex 在执行任何代码修改、重构、修复、生成组件、实现功能、调整 UI、编写测试之前，必须优先阅读并遵守本文件。  
> 如用户的明确指令与本文件冲突，以用户本轮明确指令为准；否则必须严格遵守本文件。

---

## 1. Harness Engineering 总原则

### 1.1 Role / 角色

你是本项目的移动端 App 工程师，负责将 Stitch 设计稿转化为可运行、可维护、可扩展的真实代码。

你的工作不是自由发挥产品设计，而是：

1. 严格还原既定产品方向。
2. 严格遵守三栏底部导航。
3. 严格使用简体中文界面。
4. 严格控制改动范围。
5. 保持代码清晰、稳定、安全。
6. 每次只完成一个明确任务，不做无关重构。

---

### 1.2 Mission / 项目目标

本项目是一个 **每日生活记录 App**，用于记录用户每天：

- 做了什么事情
- 今天的心情
- 开心的事情
- 不开心的事情
- 每日照片
- 每日照片封面
- 生活标签
- 每日一句话总结
- AI 温柔总结
- 时间胶囊
- 年度回忆报告

产品核心定位：

> 一个用 Emoji、日历、照片和温柔总结记录生活的私人日记 App。

它不是任务管理工具，不是社交产品，不是复杂数据面板，也不是心理诊断工具。

---

### 1.3 Product Scope / 产品范围

本项目底部导航只能有三个主页面：

1. 日历
2. 记录
3. 我的

不得新增第四个、第五个底部导航。

#### 日历页职责

日历页是历史记录中心，负责：

- Emoji 心情日历
- 按年月查看历史记录
- 点击日期查看当天预览
- 进入当天完整记录详情
- 搜索历史记录
- 月度/年度回顾入口

#### 记录页职责

记录页是每日记录入口，负责：

- 今日心情 Emoji
- 今天做了什么
- 开心的事
- 不开心的事
- 今日照片
- 每日照片封面
- 生活标签
- 每日一句话总结
- AI 温柔总结
- 时间胶囊创建入口
- 保存今日记录

#### 我的页职责

我的页是个人回顾和设置中心，负责：

- 记录天数
- 最常见心情
- 照片数量
- 常用标签
- 心情趋势图
- 生活标签统计
- 时间胶囊管理
- 年度回忆报告
- 隐私锁
- 数据导出
- 主题设置
- AI 设置

## 2. Hard Constraints / 强制限制

以下规则不可违反。

### 2.1 语言限制

所有用户可见界面文字必须使用 **简体中文**。

包括但不限于：

- 页面标题
- 按钮文字
- 标签文字
- 表单占位符
- 错误提示
- 成功提示
- 空状态文案
- 弹窗文案
- 导航文字
- 设置项文案
- 示例内容

禁止在 UI 中使用英文作为主要界面语言。

允许出现英文的情况仅限：

1. 技术变量名、代码注释、文件名。
2. API、JSON、TypeScript 类型定义。
3. 品牌名或用户明确要求保留的英文名称。
4. 图标库或第三方库本身的技术名称。

---

### 2.2 导航限制

底部导航只能包含：

```text
日历 / 记录 / 我的
```

禁止新增以下底部导航：

- 趋势
- 回忆
- 时间胶囊
- 年度报告
- 设置
- AI
- 照片
- 搜索

这些功能只能作为二级页面入口放在：

- 日历页
- 记录页
- 我的页

---

### 2.3 设计风格限制

必须保持以下 UI 风格：

```text
温柔治愈
极简
低压力
奶油白背景
低饱和配色
圆角卡片
轻微阴影
Emoji 点缀
轻手账感
Apple-like minimal
soft diary aesthetic
```

禁止把界面改成：

- 强任务管理风
- 强打卡风
- 复杂数据面板风
- 金融图表风
- 社交信息流风
- 游戏化排行榜风
- 过度拟物风
- 高饱和炫彩风

---

### 2.4 功能范围限制

未经用户明确要求，禁止主动新增以下功能：

- 账号系统
- 登录注册
- 云同步
- 社交分享
- 好友系统
- 评论系统
- 点赞系统
- 公开社区
- 复杂后端
- 支付系统
- 订阅系统
- 推送通知
- 自动读取相册
- 自动读取定位
- 自动读取联系人
- 心理诊断
- 医疗建议
- AI 聊天机器人
- 多人协作

第一版优先本地记录、本地存储、本地查看。

---

### 2.5 隐私限制

本 App 涉及日记、照片、情绪和个人生活记录，必须遵守隐私优先原则。

禁止：

1. 默认上传用户日记到云端。
2. 默认上传用户照片到云端。
3. 默认收集定位、联系人、通讯录。
4. 在客户端硬编码 API Key。
5. 在日志中打印用户日记正文。
6. 在日志中打印 AI 总结原文。
7. 在日志中打印照片路径以外的敏感信息。
8. 未经用户明确开启就调用远程 AI 服务。
9. 把私密记录用于任何分析之外的目的。

必须：

1. 默认本地优先。
2. 明确区分本地 mock AI 和真实远程 AI。
3. 真实 AI 调用必须经过安全后端代理。
4. 用户可关闭 AI 功能。
5. 后续如实现数据导出，必须由用户主动触发。

## 3. Tech Stack / 技术约束

### 3.1 推荐技术栈

默认使用：

```text
React Native
Expo
Expo Router
TypeScript
SQLite
本地图片存储
```

### 3.2 禁止随意替换技术栈

未经用户明确要求，禁止将项目改成：

- Flutter
- 原生 Android
- 原生 iOS
- Vue
- Angular
- Next.js Web App
- Electron
- 后端优先架构

如果当前仓库已经有既定技术栈，优先遵守仓库已有技术栈，不要重建项目。

---

### 3.3 TypeScript 规则

必须使用 TypeScript。

要求：

1. 禁止随意使用 `any`。
2. 需要定义清晰的类型。
3. 组件 props 必须有类型。
4. 数据库返回数据需要映射为明确类型。
5. 服务层函数需要明确返回值类型。
6. 对可能为空的数据使用安全处理。
7. 不允许用 `// @ts-ignore` 掩盖问题，除非有非常明确的理由并写注释说明。

## 4. Data Model / 数据模型约束

### 4.1 每日记录 DiaryEntry

每日记录必须围绕以下核心字段设计：

```ts
type MoodType =
  | "very_happy"
  | "happy"
  | "calm"
  | "normal"
  | "tired"
  | "anxious"
  | "sad"
  | "angry"
  | "special";

interface DiaryEntry {
  id: string;
  date: string;
  mood: MoodType;
  moodScore: number;
  whatHappened?: string;
  happyEvents: string[];
  unhappyEvents: string[];
  oneSentenceSummary?: string;
  aiSummary?: string;
  createdAt: string;
  updatedAt: string;
}
```

### 4.2 日期唯一性

每天只允许一条主记录。

如果用户保存同一天的记录：

- 如果该日期没有记录：创建新记录。
- 如果该日期已有记录：更新原记录。
- 禁止生成多个相同日期的主日记记录。

---

### 4.3 心情映射

默认心情 Emoji 和分数如下：

| MoodType | 中文名称 | Emoji | 分数 |
|---|---|---:|---:|
| very_happy | 很开心 | 😆 | 5 |
| happy | 开心 | 😊 | 4 |
| calm | 平静 | 😌 | 3.5 |
| normal | 普通 | 😐 | 3 |
| tired | 疲惫 | 😮‍💨 | 2.5 |
| anxious | 焦虑 | 😟 | 2 |
| sad | 难过 | 😢 | 1.5 |
| angry | 生气 | 😠 | 1.5 |
| special | 特别的一天 | ✨ | 5 |

禁止随意改动这些基础映射，除非用户明确要求。

---

### 4.4 照片数据

照片数据至少包括：

```ts
interface DiaryPhoto {
  id: string;
  diaryId: string;
  localUri: string;
  caption?: string;
  isCover: boolean;
  createdAt: string;
}
```

规则：

1. 每日可有多张照片。
2. 每日最多只能有一张封面照片。
3. 如果用户上传第一张照片，默认可设为封面。
4. 删除封面照片时，需要重新选择封面或清空封面状态。
5. 不得默认上传照片到远程服务器。

---

### 4.5 标签数据

标签必须是可复用数据，不要只写死在页面里。

基础标签可以包括：

```text
学习
编程
阅读
运动
散步
朋友
家人
独处
焦虑
放松
完成任务
睡眠好
睡眠差
美食
```

允许用户后续自定义标签，但第一版可以先使用固定标签。

---

### 4.6 时间胶囊数据

时间胶囊至少包括：

```ts
interface TimeCapsule {
  id: string;
  diaryId?: string;
  title: string;
  content: string;
  mood?: MoodType;
  openDate: string;
  isOpened: boolean;
  createdAt: string;
  openedAt?: string;
}
```

规则：

1. 未到 `openDate` 的时间胶囊不能展示正文。
2. 到达 `openDate` 后可以打开。
3. 打开后标记 `isOpened = true`。
4. 时间胶囊可以关联某天日记，但不能强制要求关联。

## 5. UI Rules / UI 编程规则

### 5.1 页面结构

必须保持以下页面结构：

```text
app/
  calendar
  record
  profile
  day-detail/[date]
  capsule/create
  capsule/index
  stats/mood-trend
  stats/tags
  report/yearly
  settings
```

如果项目使用不同路由结构，可以适配，但逻辑上必须保持：

- 三个主页面
- 其他为二级页面

---

### 5.2 日历页规则

日历页必须包含：

1. 当前年月。
2. 今天按钮。
3. 月份切换。
4. Emoji 心情日历。
5. 选中日期预览卡片。
6. 查看完整记录入口。

日期格子规则：

1. 有记录时显示主心情 Emoji。
2. 无记录时只显示日期。
3. 有照片时显示小图片角标。
4. 有时间胶囊时显示小信封角标。
5. 当前日期使用柔和背景。
6. 选中日期使用柔和描边或轻微高亮。

禁止把日历页改成纯列表页。

---

### 5.3 记录页规则

记录页必须包含：

1. 今日日期。
2. 今日心情选择。
3. 今天做了什么。
4. 让我开心的事。
5. 让我不开心的事。
6. 今日照片。
7. 生活标签。
8. 一句话总结。
9. AI 温柔总结。
10. 写给未来的自己入口。
11. 保存按钮。

规则：

1. 使用卡片式分区。
2. 不要做成复杂长表单。
3. 输入框文案必须温柔、自然。
4. 保存成功提示必须使用简体中文。
5. 如果该日期已有记录，必须自动加载并允许编辑。
6. 保存时同一天记录必须更新，而不是重复创建。

---

### 5.4 我的页规则

我的页必须包含：

1. 本月记录天数。
2. 最常见心情。
3. 照片数量。
4. 常用标签。
5. 心情趋势图入口。
6. 生活标签统计入口。
7. 时间胶囊入口。
8. 年度回忆报告入口。
9. 隐私锁入口。
10. 数据导出入口。
11. 主题设置入口。
12. AI 设置入口。

禁止把我的页变成单纯设置列表。

它应该是：

> 个人回顾中心 + 设置中心

---

### 5.5 图表规则

心情趋势图和标签统计必须保持柔和风格。

禁止：

- 使用过度强烈颜色。
- 做成金融数据面板。
- 使用复杂坐标轴。
- 在一屏堆太多指标。
- 使用过多图表类型。

推荐：

- 柔和折线图。
- 圆角图表卡片。
- 简单平均值。
- 温柔解释文案。
- 近 7 天、近 30 天、本月切换。

## 6. AI Summary Rules / AI 总结规则

### 6.1 AI 定位

AI 温柔总结只能作为生活记录辅助。

它不是：

- 心理咨询师
- 医生
- 诊断工具
- 情感操控工具
- 说教工具

AI 总结语气必须：

```text
温柔
克制
简洁
不说教
不夸张
不诊断
像朋友一样陪伴
```

---

### 6.2 第一版必须使用 Mock

除非用户明确要求接入真实 API，第一版 AI 总结必须使用本地 mock 实现。

Mock 函数可以根据：

- 心情
- 今天做了什么
- 开心的事
- 不开心的事
- 标签
- 一句话总结

生成温柔总结。

---

### 6.3 真实 AI API 限制

如果用户要求接入真实 AI API，必须遵守：

1. 不得在客户端硬编码 API Key。
2. 必须使用环境变量。
3. 移动端不得直接暴露密钥。
4. 推荐通过后端代理调用。
5. API 失败时回退到本地 mock。
6. 用户可以在设置中关闭 AI。
7. 不得把 AI 总结写成心理诊断。

---

### 6.4 禁止输出内容

AI 总结中禁止出现：

- “你患有……”
- “你有心理疾病……”
- “必须……”
- “你应该立刻……”
- “你的人生……”
- 夸张、恐吓、诊断、命令式表达

推荐表达：

- “今天的你经历了一些起伏。”
- “你已经认真记录了自己的感受。”
- “明天可以给自己留一点轻松的时间。”
- “这一天虽然不完美，但也有值得被看见的部分。”

## 7. Code Organization / 代码组织规则

```text
src/

  db/

  services/
  theme/
```

- TimeCapsuleCard

只放数据库相关内容，例如：

- schema
- database init

- migrations

只放类型定义。

只放业务服务，例如：

只放设计系统，例如：

### 7.3 禁止行为

1. 把所有代码写在一个页面文件里。

3. 在数据库层直接写 UI 文案。
4. 在页面中硬编码大量 mock 数据。
5. 在多个文件重复定义同一套心情映射。
6. 用复制粘贴制造多个相似组件。
7. 为了赶进度删除已有功能。
8. 未经允许重命名大量文件。
9. 未经允许改变项目技术栈。
10. 未经允许进行大规模格式化导致 diff 过大。

## 8. Workflow / Codex 工作流

每次执行任务时，必须按照以下步骤进行。

### 8.1 Understand / 理解任务

开始前先判断：

1. 用户要求实现什么？
2. 影响哪些页面？
3. 是否涉及数据库？
4. 是否涉及隐私或 AI？
5. 是否会影响已有功能？
6. 是否需要测试？

不要直接开始大范围改代码。

---

### 8.2 Inspect / 检查现有代码

修改前必须检查：

1. 项目结构。
2. 相关页面文件。
3. 相关组件。
4. 相关类型。
5. 数据库 schema。
6. 是否已有类似实现。
7. 是否已有设计规范或 DESIGN.md。

禁止在不了解现有代码的情况下新建重复实现。

---

### 8.3 Plan / 制定小计划

每次任务必须拆成小步骤。

示例：

```text
1. 检查现有记录页结构。
2. 新增 MoodEmojiPicker 组件。
3. 接入记录页状态。
4. 保存 mood 到 SQLite。
5. 添加空状态和错误处理。
6. 运行类型检查。
```

禁止一次性承诺完成超大范围任务。

---

### 8.4 Implement / 实现

实现时必须：

1. 控制改动范围。
2. 优先复用已有组件。
3. 保持设计系统一致。
4. 保持简体中文 UI。
5. 保持类型安全。
6. 保持本地优先。
7. 不做无关重构。

---

### 8.5 Verify / 验证

完成后必须尽可能运行：

```bash
npm run lint
npm run typecheck
npm test
npm run start
```

如果项目没有这些脚本：

1. 不要随意新增复杂工具链。
2. 说明项目缺少对应脚本。
3. 至少确保 TypeScript 和代码结构没有明显错误。

---

### 8.6 Summarize / 总结

完成任务后必须总结：

1. 修改了哪些文件。
2. 实现了哪些功能。
3. 如何验证。
4. 是否有未完成事项。
5. 是否需要用户手动配置。

不要只说“完成了”。

---

## 9. Change Control / 改动控制

### 9.1 小步提交原则

每次 Codex 任务只做一个明确主题。

推荐任务粒度：

- 实现 MoodEmojiPicker。
- 实现记录页保存。
- 实现日历页月份切换。
- 实现时间胶囊创建。
- 实现心情趋势图。
- 修复照片封面选择问题。

不推荐任务粒度：

- 一次性完成整个 App。
- 重构所有页面。
- 重写全部 UI。
- 同时实现数据库、AI、图表、导出、设置。

---

### 9.2 Diff 限制

除非用户要求大规模重构，否则：

1. 不要修改与任务无关的文件。
2. 不要格式化整个项目。
3. 不要删除已有功能。
4. 不要重写整个页面。
5. 不要替换设计系统。
6. 不要引入大量新依赖。

---

### 9.3 新依赖限制

新增依赖前必须判断：

1. 是否确实需要？
2. 是否可以用已有依赖完成？
3. 是否适合 Expo / React Native？
4. 是否会增加隐私风险？
5. 是否会显著增加包体积？
6. 是否有维护风险？

禁止为了简单功能引入大型依赖。

---

## 10. Error Handling / 错误处理规则

必须处理以下状态：

### 10.1 数据为空

例如：

- 某天没有记录。
- 本月没有记录。
- 没有照片。
- 没有标签统计。
- 没有时间胶囊。
- 年度报告数据不足。

必须显示温柔的简体中文空状态文案。

示例：

```text
这一天还没有记录。
还没有照片，给今天留下一张回忆吧。
这个月的记录还不多，慢慢来就好。
```

---

### 10.2 保存失败

必须显示：

```text
保存失败，请稍后再试。
```

并保留用户已输入内容，禁止清空表单。

---

### 10.3 图片权限失败

必须显示：

```text
需要相册权限才能选择照片。
```

不得崩溃。

---

### 10.4 AI 生成失败

必须显示：

```text
这次没有生成成功，可以稍后再试。
```

并允许用户手动填写。

---

## 11. Accessibility / 可访问性规则

必须尽量做到：

1. 按钮有清晰文字。
2. 图标不能单独承担含义。
3. Emoji 旁边需要有中文心情名称。
4. 颜色不能作为唯一信息来源。
5. 文字对比度足够。
6. 点击区域足够大。
7. 输入框有明确占位说明。

## 12. Testing Rules / 测试规则

### 12.1 必测场景

实现功能后至少考虑以下场景：

#### 记录页

- 新建今天记录。
- 编辑今天记录。
- 心情为空时的处理。
- 开心/不开心内容为空时的处理。
- 上传照片后保存。
- 设置照片封面。
- 添加标签。
- 生成 AI mock 总结。

#### 日历页

- 有记录的日期显示 Emoji。
- 无记录的日期不显示 Emoji。
- 切换月份。
- 点击日期显示预览。
- 点击查看完整记录。

#### 详情页

- 有记录时正常展示。
- 无记录时显示空状态。
- 点击编辑跳转记录页。

#### 时间胶囊

- 创建时间胶囊。
- 未到日期不能查看正文。
- 到日期后可以打开。
- 打开后状态变为已开启。

#### 统计

- 无数据时显示空状态。
- 有数据时显示趋势。
- 标签出现次数正确。
- 平均心情计算正确。

## 13. Copywriting Rules / 文案规则

所有用户界面文案要符合以下风格：

```text
温柔
自然
简洁
不过度鸡汤
不制造焦虑
不命令用户
```

推荐文案：

```text
今天感觉怎么样？
记录一点今天的小事吧。
给今天写一句话。
写给未来的自己。
这一天还没有记录。
慢慢来，记录本来就是一件温柔的小事。
```

禁止文案：

```text
你必须完成今日打卡。
你今天失败了。
你的情绪很糟糕。
系统检测到你状态异常。
立即改善你的生活。
```

## 14. Security Rules / 安全规则

必须遵守：

1. 不在客户端硬编码密钥。
2. 不提交 `.env`。
3. 不打印敏感日记内容。
4. 不上传用户照片，除非用户明确开启对应功能。
5. 不默认请求不必要权限。
6. 不收集超出功能需要的数据。
7. 不把 AI API Key 放进移动端包里。
8. 不在 GitHub 中提交真实用户数据。
9. 不把测试数据写成真实隐私内容。

---

## 15. Performance Rules / 性能规则

必须注意：

1. 日历页只加载当前月份必要数据。
2. 图片列表需要控制尺寸。
3. 照片墙避免一次性加载过多原图。
4. 年度报告可先展示统计，再加载照片墙。
5. 数据库查询要按日期、月份、年份过滤。
6. 不要在 render 中做大量统计计算。
7. 重复统计应放到 service 或 utils 中。

- Expo 生态兼容库

- 轻量库
- TypeScript 支持良好的库
